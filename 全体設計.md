# WImm Lana 共有ドライブ 全体設計書

> バージョン: 0.1 (ドラフト)
> 最終更新: 2026-04-22
> ステータス: レビュー待ち（メンバー全員の合意後に Phase 1 着手）

本ドキュメントは、WImm Lana バンドの共有ドライブ運用の「全体設計」をまとめたものです。
Phase 0（設計合意）→ Phase 1（骨組み構築）→ Phase 2（自動化）→ Phase 3（カレンダー連携等の拡張）の順で段階的に構築していきます。

---

## 1. 目的とスコープ

### 1.1 このドライブで扱うもの（In Scope）

- 楽曲制作のプロジェクト管理（曲ごとの進行、歌詞、デモ、リファレンス）
- リリース・イベントの進行管理（配信、MV、ライブ、プロモ等）
- 定例 MTG の議事録・決定事項・ToDo
- 上記に紐づく共通素材（アートワーク、写真、プロフィール資料など）
- 運用ルール・テンプレート本体

### 1.2 このドライブで扱わないもの（Out of Scope）

- DAWプロジェクトファイル本体（重いので別ストレージ or 別共有方法）
- 個人のメモ・アイデア倉庫（各自のローカル Obsidian Vault で管理）
- 金銭・契約関連の機密書類（必要になったら別途権限設計）

### 1.3 ゴール

- 誰が書いても **同じ構造・同じ粒度** の情報が残る
- Obsidian で **一覧性**（Dataview）と **記述の速さ**（Templater）を担保
- Cursor / Git でも編集でき、一括整形やスクリプトによる変換が可能
- なるべく **手入力を減らす**（半自動）

---

## 2. 前提と設計原則

### 2.1 使用環境

- 共有方法: Google ドライブ共有ドライブ `WImm Lana_共有ドライブ`
- 編集クライアント: Obsidian（主）、Cursor（補助・一括編集）
- メンバー: 2〜3 人（全員が全フォルダにアクセス）

### 2.2 設計原則

1. **Plain text ファースト**: ノートは全て Markdown（.md）。Obsidian でも Cursor でも扱える
2. **1 ノート 1 目的**: 議事録、楽曲、リリースなどはファイル単位で分ける
3. **フロントマターで構造化**: ステータスや担当は YAML frontmatter に入れ、Dataview で集計可能にする
4. **命名で探さない**: フォルダの場所と命名規則で機械的に特定できる
5. **番号付きフォルダ**: ソート順を安定させ、トップ階層は動かさない
6. **Google Drive 競合対策**: 同時編集しがちな索引系ノートは自動生成にする（手書きしない）

## 3. 全体フォルダ構成

Obsidian Vault のルートは **`WImm Lana_共有ドライブ` 直下** とします（= 共有ドライブそのものが 1 つの Vault）。
こうすることで、Wiki リンク・Dataview・テンプレートが全フォルダ横断で効くようになります。

```
WImm Lana_共有ドライブ/
├── .obsidian/                    # Obsidian 設定（共有するプラグインとテーマ）
├── 00_運用/                      # 運用ルール・設計書・ガイドライン
│   ├── README.md                 # まず読むページ
│   ├── 全体設計.md              # 本ドキュメント
│   ├── 運用ルール.md
│   ├── 命名規則.md
│   └── 自動化メモ.md
├── 10_Templates/                 # Obsidian / Cursor 用テンプレート本体
│   ├── 楽曲.md
│   ├── リリース.md
│   ├── 議事録.md
│   └── タスク.md
├── 20_楽曲制作/                  # 楽曲単位のプロジェクト管理
│   ├── _Index.md                # Dataview による一覧
│   └── <楽曲ID>_<仮題>/
│       ├── 概要.md              # ステータス・担当・進行
│       ├── 歌詞.md
│       ├── 構成メモ.md
│       ├── デモ/                # mp3 / wav（軽いもののみ。重いものはリンクで）
│       └── リファレンス/
├── 30_リリース_イベント/         # リリース／ライブ／イベント単位
│   ├── _Index.md
│   └── <YYYYMMDD>_<案件名>/
│       ├── 概要.md              # ステータス・KPI・関係者
│       ├── タスク.md            # ガント的 ToDo
│       ├── 素材/                # ジャケ、プレス用写真など
│       └── 参考資料/
├── 40_MTG議事録/                 # 定例・臨時 MTG
│   ├── _Index.md
│   └── <YYYY>/
│       └── <YYYY-MM-DD>_<MTG種別>.md
├── 50_共有素材/                  # 常時参照するアセット
│   ├── アートワーク/
│   ├── 写真/
│   ├── プロフィール文/
│   └── SNS投稿素材/
├── 60_外部資料/                  # 受領した資料・契約参考・議事録
│   └── <YYYY>/
└── 90_アーカイブ/                # 完了・凍結案件
    ├── 楽曲/
    ├── リリース_イベント/
    └── MTG/
```

### 3.1 設計意図

- **10 番台 = 道具**（テンプレ）、**20〜40 = 中核**（ノート）、**50〜60 = 素材**、**90 = 終了後**
- 番号は 10 の倍数で空け、後から分野を追加できるようにしておく
- 各フォルダの `_Index.md` は Dataview で自動生成（手書き更新しない）
- 完了した案件は **削除せず 90_アーカイブ に移動**。検索性を保つ

## 4. 命名規則

### 4.1 フォルダ / ファイル名

| 対象 | 書式 | 例 |
| --- | --- | --- |
| トップ階層 | `<番号>_<カテゴリ>` | `20_楽曲制作` |
| 楽曲フォルダ | `<楽曲ID>_<仮題>` | `S003_夜明け前` |
| リリース／イベント | `<YYYYMMDD>_<案件名>` | `20260801_1stシングル配信` |
| 議事録 | `<YYYY-MM-DD>_<MTG種別>.md` | `2026-04-22_定例.md` |
| テンプレ | カテゴリ名そのまま | `楽曲.md` |

- 楽曲 ID は `S001` から連番。命名の衝突を避けるため番号必須
- 日付は `YYYY-MM-DD` または `YYYYMMDD` に統一（日本語/スラッシュ禁止）
- スペースは使わず、区切りは `_`（アンダースコア）
- 絵文字はファイル名に入れない（Google Drive との同期で壊れやすいため）

### 4.2 ステータス語彙（統一）

楽曲・リリース・タスクで同じ語彙を使います。

- `idea` アイデア段階
- `wip` 作業中
- `review` レビュー待ち
- `done` 完了
- `archived` 凍結／棚上げ

## 5. 運用ルール

### 5.1 書き込みのルール

- ノートの先頭には **必ず YAML frontmatter** を入れる（テンプレが自動挿入）
- 迷ったら `00_運用/README.md` を読む
- 新しいフォルダを作る前に、既存フォルダで表現できないかを確認
- テンプレ（`10_Templates/`）以外の場所にテンプレ用ファイルを置かない

### 5.2 ファイル役割分担

| 作業 | 主ツール | 用途 |
| --- | --- | --- |
| ノートの新規作成 | Obsidian | Templater/QuickAdd から開始 |
| 議事録の速記 | Obsidian | ライブ編集中のフォーマット補助が強い |
| 大量ファイルのリネーム / 一括置換 | Cursor | 命名規則変更、タグ一括修正 |
| スクリプト開発 | Cursor | Templater 用 JS、整形スクリプト |
| ビューの確認 | Obsidian | Dataview ダッシュボード |

### 5.3 Google Drive 同期での競合対策

- 同じファイルを同時編集しない（特に `_Index.md` は自動生成に寄せる）
- `.obsidian/workspace*.json` は各自の画面状態で競合しやすいため、**共有しない**（`.gitignore` 相当の除外リストに入れる想定）
- 競合ファイル（末尾に `(競合 YYYY-MM-DD)` が付くもの）を見つけたら、手元で統合してから片方を削除

### 5.4 権限と削除

- 共有ドライブの権限は「コンテンツ管理者」以上をメンバーに付与
- **削除は即実行しない**。まず `90_アーカイブ/` に移動して 1 ヶ月様子を見る

## 6. 自動化（半自動）の設計

レベルは **Obsidian/Cursor スクリプトで半自動** を採用。具体的には以下を組み合わせます。

### 6.1 使うプラグイン / 機能

| プラグイン | 役割 |
| --- | --- |
| Templater | フロントマター・日付・ファイル名の自動挿入 |
| QuickAdd | 「新しい楽曲を作る」「議事録を作る」をワンキーで実行 |
| Dataview | 楽曲一覧・タスク抽出・議事録インデックスの自動生成 |
| Periodic Notes | 週次ノート（リリース進捗サマリ等に使える） |
| Tag Wrangler | タグの統制 |

### 6.2 自動化したい典型操作

1. **新しい楽曲を登録**: QuickAdd → 曲名入力 → `20_楽曲制作/<自動採番ID>_<曲名>/` を作成し、概要・歌詞・構成メモの 3 ノートをテンプレから生成
2. **新しい議事録**: QuickAdd → 種別選択 → `40_MTG議事録/<YYYY>/<YYYY-MM-DD>_<種別>.md` を生成し、前回議事録の ToDo を引き継ぎ
3. **楽曲一覧（Dataview）**: frontmatter の status を集計して `20_楽曲制作/_Index.md` に `wip` → `review` → `idea` の順で並べる
4. **横断 ToDo ビュー**: 各ノートの `- [ ]` タスクを担当者別に抽出して `00_運用/ToDo.md` に表示
5. **アーカイブスクリプト（Cursor）**: frontmatter の `archived: true` が付いたフォルダを `90_アーカイブ/` に移動するシェル／Node スクリプト

### 6.3 Cursor / Git 併用（任意）

- 共有ドライブとは別に **GitHub プライベートリポジトリ** にミラーする運用も可能（履歴管理のため）。ただし Phase 1 では入れない
- 大規模な構造変更・命名規則変更は Cursor で一括 sed/rename する前提

### 6.4 将来: Google カレンダー連携（Phase 3）

Obsidian だけでは難しいので、Phase 3 で以下の選択肢を検討：

- Google Apps Script で frontmatter の `due` / `release_date` を読み取り、Google カレンダーにイベント作成
- Obsidian の Google Calendar プラグインで片方向（読み取り）のみ連携
- Notion Calendar / Amie 等の外部サービス連携

Phase 3 で改めて選定。Phase 1–2 は踏み込まない。

## 7. テンプレート計画

Phase 1 で以下 4 種を作成します。各テンプレは frontmatter + 見出しの雛形を含みます。

### 7.1 楽曲.md（例示）

```yaml
---
type: song
song_id: S001
title: 仮題
status: idea            # idea | wip | review | done | archived
owner:                   # 主担当
members: []              # 参加メンバー
bpm:
key:
created: {{date}}
updated: {{date}}
tags: [song]
---
```

見出し: `## 概要` / `## 構成` / `## コード・メロ` / `## 歌詞` / `## デモ／ファイル` / `## ToDo` / `## メモ`

### 7.2 リリース.md

frontmatter: `type: release`, `release_date`, `channels`（配信先）, `status`, `owner`
見出し: `## 概要` / `## マイルストーン` / `## タスク` / `## 関係者` / `## 素材チェックリスト` / `## リスク`

### 7.3 議事録.md

frontmatter: `type: meeting`, `date`, `kind`（定例/制作/リリース等）, `attendees`
見出し: `## アジェンダ` / `## 議論メモ` / `## 決定事項` / `## ToDo`（担当・期限つきチェックボックス） / `## 次回アジェンダ候補`

### 7.4 タスク.md（単体タスクを独立ノートにするとき用）

frontmatter: `type: task`, `status`, `due`, `owner`, `project`（楽曲 ID やリリース ID）
見出し: `## 背景` / `## 完了条件` / `## メモ`

## 8. ダッシュボード計画

`00_運用/README.md` をホームにして、以下のビューを Dataview で配置します。

- 直近 1 週間の議事録（ToDo 抽出付き）
- ステータス別 楽曲一覧（wip → review → idea）
- 直近 90 日のリリース／イベント
- 未完了 ToDo（担当別にセクション分け）

## 9. 段階的ロードマップ

### Phase 0 — 設計合意（今ここ）

- 本ドキュメントをメンバー全員がレビュー
- スコープ・フォルダ構成・命名規則に合意
- 不明点・異論は `00_運用/全体設計.md` に直接コメントで追記

### Phase 1 — 骨組み構築（1〜2 日）

1. `00_運用/` の `README.md` / `運用ルール.md` / `命名規則.md` を作成
2. `10_Templates/` に 4 種のテンプレを作成
3. `20_楽曲制作/` `30_リリース_イベント/` `40_MTG議事録/` `50_共有素材/` `90_アーカイブ/` を空で作成
4. 各フォルダの `_Index.md` を Dataview クエリのみで設置
5. 既存の曲・議事録があれば、新ルールで 1〜2 件だけ移行してみる

### Phase 2 — 半自動化（1 週間）

1. Obsidian プラグイン導入: Templater / QuickAdd / Dataview / Periodic Notes
2. QuickAdd のマクロを 3 本作成: 「新しい楽曲」「新しい議事録」「新しいリリース」
3. ToDo 横断ビュー（Dataview クエリ）を `00_運用/README.md` に載せる
4. アーカイブ用の Node/Bash スクリプトを `00_運用/自動化メモ.md` に添えて保存

### Phase 3 — 拡張（必要になったら）

- Google カレンダー連携の選定・導入
- Git ミラーによる履歴管理
- 外部ライター／エンジニアを招く場合の権限フォルダ分離
- リリース KPI ダッシュボード（配信数、SNS 反応など）
